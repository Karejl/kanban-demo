# Справочник Git для джуна

## 1. Быстрый старт

Минимальный цикл работы с git — 4 команды:

1. **Инициализация**: `git init` — создаёт репозиторий в папке проекта
2. **Добавление файлов**: `git add .` — готовит все изменённые файлы к коммиту
3. **Коммит**: `git commit -m "описание"` — сохраняет снимок состояния
4. **Проверка**: `git status` и `git log` — что изменилось и история коммитов

```bash
git init                        # создать репозиторий (папка .git с историей)
git add .                       # добавить все изменения в staging area
git add file.html               # добавить только один файл
git commit -m "feat: kanban"    # зафиксировать снимок
git status                      # что изменилось, что готово к коммиту
git log --oneline               # история коммитов (кратко)
```

## 2. .gitignore — игнорирование файлов

### Зачем
Файл `.gitignore` указывает git, какие файлы и папки **не нужно отслеживать и коммитить**.

### Что игнорировать

| Что | Почему |
|---|---|
| `node_modules/` | Зависимости — ставятся через `npm install` |
| `dist/`, `build/` | Сборка — генерируется автоматически |
| `*.log` | Логи — временные, каждый раз новые |
| `.env` | **Секреты (ключи, пароли) — нельзя в репозиторий!** |
| `*.tmp`, `*.bak` | Временные файлы |
| `.DS_Store` | Системный мусор macOS |
| `thumbs.db` | Системный мусор Windows |
| `.vscode/`, `.idea/` | Настройки редактора — у каждого свои |

### Как создать
Файл `.gitignore` кладётся в корень проекта (рядом с `.git/`):

```gitignore
# Папки (слэш на конце = папка)
node_modules/
dist/
build/
.vscode/

# Файлы по расширению
*.log
*.tmp
*.bak

# Конкретный файл
.env
secrets.json

# Исключение — НЕ игнорировать (восклицательный знак)
!important.txt

# Игнорировать всё в папке, кроме одного файла
src/temp/*
!src/temp/keep.me

# Файл в корне проекта (слэш в начале = только корень)
/config.local.js

# Файл на любой глубине внутри dir
src/**/debug.js
```

### Синтаксис

| Правило | Значение |
|---|---|
| `file.txt` | Игнорировать файл в любой папке |
| `/file.txt` | Только в корне проекта |
| `dir/` | Игнорировать всю папку |
| `*.log` | Все файлы с расширением .log |
| `!file.txt` | Исключение — НЕ игнорировать |
| `dir/**/file` | Файл на любой глубине внутри dir |
| `# текст` | Комментарий |

### Команды

```bash
# Проверить, игнорируется ли файл
git check-ignore -v file.txt

# Посмотреть, что игнорируется
git status --ignored

# Перестать отслеживать уже закоммиченный файл (оставить на диске)
git rm --cached file.txt
git rm --cached -r dir/         # то же для папки
git commit -m "stop tracking file.txt"
```

**Важно**: `.gitignore` не удаляет уже отслеживаемые файлы. Если файл уже в коммите — добавление в `.gitignore` не сработает. Нужно `git rm --cached`.

### Готовые шаблоны
Не пиши `.gitignore` с нуля — возьми готовый шаблон:
- https://github.com/github/gitignore — шаблоны для любого языка/фреймворка

## 3. Хеши коммитов

### Что такое хеш коммита
Уникальный идентификатор коммита (40 символов). Достаточно первых 7 символов, например `c601178`.

```bash
git log --oneline              # краткие хеши + сообщения
git log --format="%H %s" -3    # полные хеши + сообщения
git rev-parse HEAD             # полный хеш последнего коммита
git show a1b2c3d               # посмотреть содержимое коммита
git show --stat a1b2c3d        # какие файлы изменились в коммите
```

## 4. Проверка состояния

```bash
git status                     # что изменилось, что готово к коммиту
git diff                       # какие именно изменения (незакоммиченные)
git diff --staged              # что попадёт в коммит (уже добавленное)
git log --oneline -5           # последние 5 коммитов
git branch -a                  # все ветки
```

## 5. Коммиты

### Создать коммит
```bash
git add filename.html          # добавить конкретный файл
git commit -m "описание"      # создать коммит
```

### Правила коммитов
- Коммить после каждого завершённого изменения
- НЕ накапливать всё в одном коммите
- Описание — что и зачем, а не что сделал
- Проверяй `git diff` перед коммитом

## 6. Откат изменений

### Незакоммиченные изменения
```bash
git checkout -- filename.html  # вернуть файл к последнему коммиту
git checkout -- .              # вернуть все файлы
```
Внимание: изменения пропадут безвозвратно!

### Закоммиченные, но НЕ запушенные
```bash
git reset --soft HEAD~1        # удалить коммит, сохранить изменения в staged
git reset --mixed HEAD~1       # удалить коммит, сохранить изменения (unstaged)
git reset --hard HEAD~1        # удалить коммит И изменения полностью (опасно!)
```

### Уже запушенные (безопасно)
```bash
git revert a1b2c3d             # создать новый коммит, отменяющий a1b2c3d
git revert HEAD                # отменить последний коммит
```
Это самый безопасный способ — история сохраняется.

### Восстановить удалённый файл
```bash
git checkout a1b2c3d -- filename.html    # взять файл из конкретного коммита
```

## 7. Восстановление потерянных коммитов

```bash
git reflog                     # история всех действий (даже удалённых коммитов)
git reset --hard HEAD@{1}      # вернуться к предыдущему состоянию
git checkout -b recovery a1b2c3d  # создать ветку из потерянного коммита
```

## 8. Ветки

```bash
git checkout -b feature-name   # создать и перейти в новую ветку
git checkout master            # вернуться в основную ветку
git branch -D feature-name     # удалить ветку
git merge feature-name         # слить ветку в текущую
```

Ветки — для экспериментов. Если не получилось — просто удали ветку.

---

## 9. Stash (временное хранение)

```bash
git stash                      # спрятать незакоммиченные изменения
git stash pop                  # вернуть изменения
git stash list                 # посмотреть список stash
git stash drop                 # удалить stash
```

Полезно когда нужно срочно переключиться, но изменения ещё не готовы к коммиту.

## 10. Удалённые репозитории (GitHub)

### Подключить
```bash
git remote add origin https://github.com/ЛОГИН/репо.git   # привязать удалённый репо
git remote -v                                            # проверить связь
```

### Push (отправить на GitHub)
```bash
git push -u origin main          # первый push (запомнить связь ветки)
git push                         # последующие push
git push -u origin feature-name  # запушить другую ветку
```
Флаг `-u` (= `--set-upstream`) нужен только один раз — потом достаточно `git push`.

### Pull (получить с GitHub)
```bash
git pull origin main             # скачать и влить изменения
git pull                         # если ранее был -u
git fetch                        # только скачать, не вливать (безопаснее)
```
`fetch` — посмотреть что изменилось, `pull` — сразу применить.

### Отключиться
```bash
git remote remove origin         # отключить удалённый репо
```
Локальные файлы и коммиты остаются.

### Авторизация
GitHub требует **Personal Access Token** вместо пароля:
Settings → Developer settings → Personal access tokens → Generate new token

### Конфликты при push
```bash
git pull origin main --rebase    # подтянуть + наложить свои коммиты сверху
git push                         # затем повторить push
```

### Переименовать ветку master → main
```bash
git branch -M main               # переименовать текущую ветку
git push -u origin main          # запушить под новым именем
```

## 11. Восстановление удалённой ветки

```bash
git reflog                                  # найти хеш последнего коммита ветки
git checkout -b branch-name a1b2c3d         # пересоздать ветку из найденного хеша
git branch branch-name HEAD@{1}            # быстро — если ветка была удалена недавно
```
`reflog` хранит историю ~90 дней, после этого восстановить невозможно.

## 12. Перенос изменений между ветками

### Merge (слияние)
```bash
git checkout main               # перейти в целевую ветку
git merge feature-name          # влить изменения
```
Создаёт merge-коммит, сохраняет всю историю.

### Rebase (перемещение)
```bash
git checkout feature-name       # перейти в исходную ветку
git rebase main                 # переиграть коммиты поверх main
git checkout main
git merge feature-name          # fast-forward слияние
```
Линейная история. **Не применять к запушенным веткам!**

### Cherry-pick (один коммит)
```bash
git checkout main
git cherry-pick a1b2c3d         # перенести конкретный коммит
git cherry-pick a1b2c3d e4f5g6h # несколько коммитов
```

### Один файл из другой ветки
```bash
git checkout feature-name -- file.html   # взять файл без слияния
```

## 14. Шпаргалка: что выбрать

| Ситуация | Команда | Опасность |
|---|---|---|
| Изменил файл, хочу вернуть | `git checkout -- file` | Средняя |
| Удалил файл случайно | `git checkout hash -- file` | Низкая |
| Хочу отменить последний коммит (локально) | `git reset --soft HEAD~1` | Низкая |
| Хочу удалить коммит и изменения | `git reset --hard HEAD~1` | Высокая |
| Хочу отменить запушенный коммит | `git revert hash` | Низкая |
| Потерял коммит | `git reflog` → `git reset --hard HEAD@{n}` | Средняя |
| Нужно поэкспериментировать | Создать ветку | Низкая |
| Нужен перерыв, изменения не готовы | `git stash` | Низкая |

---

## 15. Золотые правила

1. **`git revert` — безопасно, `git reset --hard` — опасно**
2. Коммить часто, мелкими порциями
3. Проверять `git diff` перед коммитом
4. Эксперименты — только в ветках
5. Если сомневаешься — `git stash`, а не `git reset --hard`
6. Запушенное можно отменить только через `git revert`
7. `git reflog` — спасательная шлюпка, почти всегда может помочь
